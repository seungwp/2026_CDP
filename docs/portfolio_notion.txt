================================================================
SafeCar — 자율주행 안전 감독(Fail-safe) 시스템
2026 융합 Capstone Design Project (영남대학교)
================================================================

운전자 이상 · 전방 장애물 · 연산장치 고장까지 대응해
어떤 상황에서도 스스로 안전하게 멈추는 ROS 2 자율주행 플랫폼

개발 기간   : 2026.05 ~ 2026.09
팀 구성     : 4인 (인지 / 제어·통합 / 통신 / 대시보드)
담당 역할   : 제어 · 시스템 통합 (safecar_control, safecar_bringup)
기반 플랫폼 : STELLA N1 (Raspberry Pi 5 + Hailo-8 AI HAT + YDLIDAR X4)
미들웨어    : ROS 2 Jazzy


----------------------------------------------------------------
1. 프로젝트 소개
----------------------------------------------------------------

■ 문제의식
자율주행 연구는 "얼마나 잘 달리는가"에 집중. 그러나 실제 사고는
'예외 상황에서 시스템이 어떻게 무너지는가'에서 발생.

  ① 운전자가 주행 중 건강 이상을 일으키면?
  ② 전방에 갑자기 장애물이 나타나면?
  ③ 자율주행을 계산하던 컴퓨터 자체가 멈추면?

→ 일반 자율주행 스택 위에 안전 감독(Supervisor) 레이어를 얹은 프로젝트

■ 핵심 아이디어 : 단일 안전 게이트 + 하드웨어 이중화
· 차체 위에 [인지(카메라·NPU)] → [판단(제어)] → [통신(생체센서)] 3계층 추가
· 바퀴로 나가는 주행 명령(/cmd_vel)은 단 하나의 안전 게이트만 통과
  → 어떤 노드도 바퀴에 직접 명령 불가, 모든 명령은 게이트 검사를 거침
· 게이트가 도는 라즈베리파이 자체의 고장 대비 → STM32 CAN 워치독을 최후 방어선으로 배치
  heartbeat가 끊기면 STM32가 CAN으로 직접 차량 정지

■ 판단 우선순위 (decision_maker)
  1순위  전방 장애물     → 여유공간 확인 → AVOID(회피) / EMERGENCY_BRAKE(급제동)
  2순위  운전자 생체 이상 → MRM_PULL_OVER (감속 + 우측 갓길 이동 후 정차)
  3순위  정상            → 주행 명령 통과 (1초간 명령 끊기면 자동 정지)
  최후   파이 고장       → STM32 CAN 워치독이 직접 정지

■ 시스템 흐름
  카메라 ─┬→ 차선 인식(OpenCV) ─→ 차선 추종 주행 ─┐
          └→ 장애물 인식(Hailo NPU) ──────────────┤
  생체신호 ────────────────────────────────────────┤
                                                   ↓
                                        [ 안전 게이트 ] ──/cmd_vel──→ 차체·모터
                                                   ┆ heartbeat
                                        [ STM32 워치독 ] ┄신호 끊기면 직접 정지┄→ 차체

■ 결과
  · 실내 트랙에서 OpenCV 차선 추종 자율주행 실차 end-to-end 검증
  · Hailo-8 NPU 실시간 객체 인식(약 26Hz) → 장애물 감지 → 급제동 동작 확인
  · 생체 이상 신호 → 우측 조향 후 정차(MRM) 로직 구현
  · 안전 게이트 전 경로(통과 / 오버라이드 / 타임아웃 정지) 실기 검증 완료

2. 기술 스택
----------------------------------------------------------------

[ 미들웨어 / 빌드 ]
  ROS 2 Jazzy, colcon, launch(Python), 커스텀 msg 패키지(safecar_msgs)

[ 언어 ]
  Python  — 인지 · 판단 · 제어 · 통신 노드
  C++     — 차체 드라이버 (모터 · IMU · LiDAR)

[ 비전 / AI ]

  Hailo-8 NPU + YOLOv8n(.hef) — 실시간 객체 인식 (person/car/bus/bicycle 등)
  GStreamer — Hailo 추론 파이프라인

[ 제어 ]
  P/D 기반 조향 (lane offset → angular.z)
  상태 기반 판단 (NORMAL / AVOID / EMERGENCY_BRAKE / MRM_PULL_OVER)
  Dead-man 타임아웃, 명령 단일화 게이트 패턴

[ 하드웨어 / 임베디드 ]
  Raspberry Pi 5, Hailo-8 AI HAT
  Camera Module 3 (imx708, CSI), YDLIDAR X4
  STM32 + CAN 통신 (하드웨어 워치독)

[ 개발 환경 / 협업 ]
  Ubuntu 24.04, SSH 원격 실차 디버깅, Ubuntu VM 원격 관제 PC
  Git / GitHub (파트별 패키지 분리, OWNER 기반 소유권 관리)
  rqt, rviz2, ros2 topic/bag CLI 실차 검증

3. KPT 회고
----------------------------------------------------------------

■ Keep

· 안전을 '기능'이 아니라 '구조'로 설계
  긴급정지를 노드마다 흩어두지 않고 /cmd_vel 발행 권한을 한 노드로 통합.
  차가 왜 그렇게 움직였는지 항상 한 곳에서 설명 가능, 기능 추가에도 안전 로직이 흔들리지 않음.

· 실차 검증을 미루지 않음
  벤치에서 통과해도 트랙에 올리면 항상 새 문제 발생
  (바닥 반사광을 차선으로 오인, 카메라 각도 때문에 검출선이 화면 밖으로 이탈 등)
  → "책상에서 되는 것과 바닥에서 되는 것은 다른 문제"

· 실패 상황을 기본값으로
  차선 놓치면 정지, 명령 1초 끊기면 정지, 신호 없으면 정지.
  아무 일도 없을 때의 기본 동작을 항상 안전한 쪽으로 설계.

· 코드보다 토픽 계약을 먼저 합의
  4인 동시 개발에도 충돌이 거의 없던 이유. 무엇을 주고받을지 문서로 선확정.


■ Problem

· 하드웨어 문제를 소프트웨어 문제로 오진
  차선 인식 불안정의 원인은 알고리즘이 아니라 카메라 장착 각도와 바닥 재질.
  코드부터 의심하느라 시간 소모.

· 로그 없이 감으로 튜닝한 초반
  파라미터를 바꿔가며 반복 주행 → 무엇이 개선을 만들었는지 재현 불가.
  rosbag 기록을 처음부터 습관화했어야 함.

· 상태 전이 견고성을 뒤늦게 인지
  센서 값이 한 번 튀면 상태가 즉시 복귀 → 판단 깜빡임.
  래칭과 입력 staleness 처리를 설계 단계에 넣었어야 함.

· 급제동이 말 그대로 '급'제동
  속도를 즉시 0으로 꽂아 차체 흔들림이 큼. 감속 램프 필요.

· 전력 마진 확인이 늦음
  배터리 구동 시 전압 여유가 거의 없음을 개발 중반에야 측정으로 확인.
  하드웨어 제약은 초기에 수치로 못 박아야 함.


■ Try

· rosbag 상시 기록 + 정량 지표(중심 이탈량, 조향 변화율)로 튜닝 결과 비교
· LiDAR /scan 근접 정지 및 여유공간 판단 → '정지'에서 '회피 주행'으로 확장
· 판단 로직 하드닝 — 상태 래칭, 감속 램프, 입력 staleness 타임아웃
· 실제 갓길 대피(MRM) 완성 — 정지 중심 로직을 LiDAR 기반 우측 이동으로 대체
· STM32 CAN 워치독 완성 및 '파이 전원 차단' 시나리오 실차 검증
· 노드 단위 자동 테스트 도입 → 실차 검증 전에 걸러낼 문제는 코드에서 차단
· 튜닝 상수(HSV 범위, 조향 게인)를 코드 상수 → 파라미터 파일로 완전 분리

4. 한 줄 정리
----------------------------------------------------------------
"잘 달리게 만드는 것보다, 무엇이 고장 나도 안전하게 멈추게 만드는 것이 훨씬 어렵다."
자율주행의 난이도는 정상 동작이 아니라 예외 처리에 있음을 체감한 프로젝트.
